# 実績

## BtoC向けメディアサイトの機能改善運用
### チームでの役割
フロントエンドエンジニア

### チームの編成
プロジェクトリーダー兼フロントエンドエンジニア1人、デザイナー1人、バックエンドエンジニア2人

### 使用した技術・バージョン
- Next.js（App Router）、Vercel、GraphQL、Editor.js
- 担当外：AWS

### サービス固有の技術的特徴
- バックエンドがデータ基盤を整え、フロントエンド側はGraphQLを経由して実装
- 週単位レベルでABテストを実施
- 継続的なコスト削減やパフォーマンス改善
- Next.jsベースの管理画面から記事や関連データを管理

### チームの課題と自身が工夫したこと
- スピード重視で開発されていたので、JSDocなどでコメントを充実させたり、小さな単位でリファクタリングをして、継続的な改善に耐えられる環境を意識していました。
- Next.jsやVercelなど経験が不足していたので、生成AIに叩き台を作ってもらい、公式ドキュメントを元にリファクタリングをしながら開発しました。
- Vercelのサポート担当者と連携して、コスト削減や新規機能を使った改善提案をしていました。
- QAエンジニアがいなかったので、プルリクエストのレビュー時にテストケースを書きながら品質を担保。改善点や抜け漏れなどの提案や指摘を日常的におこない、お客様からも評価いただきました。

## BtoC向けサービスサイトのリニューアル
### チームでの役割
フロントエンドエンジニア

### チームの編成
プロジェクトリーダー兼フロントエンドエンジニア1人、フロントエンドエンジニア1人、デザイナー1人、ディレクター1人、PM1人、インフラエンジニア1人

### 使用した技術・バージョン
- Next.js（App Router）、Vercel、microCMS

### サービス固有の技術的特徴
- On-Demand ISRを中心にしたレンダリング設計
- VercelのEnterpriseプランを採用

### チームの課題と自身が工夫したこと
- WordPressからの脱却とモダンな開発を社内に導入することを目的に、Next.js・microCMS・Vercelの技術スタックを提案。予算内に収めたりセキュリティ面の懸念を払拭するために、調査をしながら根拠のある提案ができるようにしました。
- 保守とサイト運用、インフラが別々の部署になるため、それぞれが単独で動けるように設計。保守とインフラが別部署になり、現状では連携がスムーズではないため、AWSで構築するよりもVercelを使用したほうが円滑になると考えて提案しています。
- デザイン案から開発要件を進めるかたちになり、サイト運用を担当する部署にもヒアリングをしながら、実装要件を整理・提案しました。
- お客さまとVercelのサポート担当者との間に入ってVercelのEnterpriseプランの導入を支援しました。

## BtoC向けサービスサイトの機能改善
### チームでの役割
ディレクター兼フロントエンドエンジニア

### チームの編成
ディレクター6人、エンジニア6人

### 使用した技術・バージョン
- Vue.js（v2）、gulp、WebRelease2

### サービス固有の技術的特徴
- スクラム+DevOpsな運用フロー
- Excelベースの仕様書やテスト設計書などのドキュメント
- 階層構造が深く、MTGや承認フローが多い開発体制
- インシデントが大きな機会損失になりやすいサービス規模の大きさ
- サービスにおけるドメイン知識の必要性とページ構成や出しわけの複雑さ
- レガシーな環境と手作業の多さを起因とする問題の発生

### チームの課題と自身が工夫したこと
- MTGとその参加人数が多いため、議題の整理を厳密におこない、効率的に結論に到達できるようにしました。
- MTGや議論の進め方の改善提案をして、作業に時間を使えるようにしました。
- 真因にまで迫れていない報告が多いため、改善提案をしたり、担当者と一緒に改善方法を検討しました。
